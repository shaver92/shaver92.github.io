Harness Engineering 实现 — GStack

Harness Engineering 实现 — GStack

概述

GStack 是 Y Combinator CEO Garry Tan 开源的一套 AI 辅助软件工程工作流。它将 Claude Code 变成一个虚拟工程团队，包含 47 个专业角色（CEO 评审、工程经理、设计师、QA、安全官、发布工程师等），每个角色都是一个 Markdown 技能文件（SKILL.md）。

  • 源码仓库：github.com/garrytan/gstack
  • 本地路径：/Users/shaver/claude_projects/GStack
  • 许可证：MIT，永久免费

核心理念

Garry Tan 用 GStack 实现了惊人的生产率飞跃：

2026 年 vs 2013 年：逻辑代码变更量是当年的 ~810 倍，2026 年前 4 个月产出已经是 2013 年全年的 240 倍。一个人 + AI 做到了以前 20 人团队的产出。

三大原则（Builder Ethos）

  1. Boil the Lake（做完整） — AI 让”做完整”的边际成本趋近于零。90% 和 100% 之间就差几秒，永远选 100%。
  2. Search Before Building（先搜索再构建） — 三层次知识：经典方案 → 流行趋势 → 第一性原理。避免重新发明轮子，但也不盲从。
  3. User Sovereignty（用户主权） — AI 建议，人类决策。这个规则高于一切。

参考源码文件：ETHOS.md，ARCHITECTURE.md

源码结构

关键架构：浏览器守护进程

首次调用 3s 启动 Chromium，后续每次 ~100-200ms。技术选型：Bun 编译为单二进制文件（58MB），内置 SQLite 用于读取浏览器 Cookie 数据库。

详见源码：ARCHITECTURE.md 第 5-50 行，browse/src/ 目录

GStack 与 Harness Engineering 的关系

GStack 不是传统意义上的 “Harness Engineering 框架”。它在三个层面上实现了 Harness 理念：

层面 GStack 中的实现 源码位置
Browser Harness (浏览器操控层) Playwright + Chromium 守护进程，持久化会话，100ms 命令延迟 browse/src/
Workflow Harness (工作流编排层) 47 个 SKILL.md，每个是一个专业工程角色，输入输出彼此衔接 各技能目录下的 SKILL.md
Agent Harness (Agent 协调层) /pair-agent 安全地让多个 AI Agent 共享同一浏览器会话 pair-agent/SKILL.md

核心洞察： GStack 的”Harness”就是那套使单个开发者能指挥整个 AI 团队的流程骨架。浏览器是可插拔的——你有自己的 browser harness 或 BAI 方案，依然可以使用 GStack 的其他 46 个技能。两者形成互补而非绑定关系。

Harness Engineering理论

1 四个核心支柱-理论内核

1.1 代码库即真相源（Codebase as Source of Truth）

类似Claude.md或是Agent.md，按需进行获取，提供类似的是一份导航图

上面md文档，可以借助AI工具进行全文阅读总结后生成，对于这种单任务，其准确性还是很高的

💡 例子：Agent 看不到的东西，就等于不存在。

团队在 Slack 里讨论决定”所有 API 接口统一用蛇形命名”。Agent 不知道这件事，继续生成驼峰命名的接口——因为那条 Slack 消息不在仓库的任何文件里。正确的做法是把这条规范写进 docs/conventions.md，AGENTS.md 再指向它。这样 Agent 每次启动都会读到，永远遵守。

1.2 机械化架构约束（Mechanized Architectural Constraints）

采用Hooks进行类似宪法性质的约束，Claude.md可以理解是建议，但是Hooks是强制性类似法律的法律的约束
对应有CLI Hooks和Command Hooks两种类型

1.2.1 CLI Hooks

每次在执行Edit/Write命令后 就需要进行格式化

💡 防火墙例子：阻止 Agent 执行危险命令。

配套脚本 .claude/hooks/block-danger.sh：

Agent 执行任何 Bash 命令前都会经过这道门禁。想 rm -rf /？直接拦截，不可能绕过。

1.2.2 Command Hooks

需要手动进行比如 /review 对当前的代码进行review

1.3 反馈循环（Feedback Loops）– 核心点

反馈循环是串联 1.1（真相源）和 1.2（机械化约束）的运转引擎。没有它，规则写完就烂在那里，Agent 该犯还是犯。有了它，每踩一次坑，全团队的 Agent 集体免疫：

Agent 犯错 → 分析根因 → 写成规则 → 加入 AGENTS.md + CLI Hook
→ 下次 Agent 启动/执行时自动读取 → 永不再犯

例子：Agent 反复把 API Key 写进代码

第 1 次犯错：Agent 在 config.ts 里硬编码了一行 apiKey = "sk-xxxxxxxx"。代码评审才发现。

第 2 次犯错：换了个文件，Agent 又干了一次。你意识到光靠人说不行。

启动反馈循环：

  1. 写成规则 → 在 docs/conventions.md 加一条：”API 密钥必须从环境变量读取，禁止硬编码”
  2. 加入 AGENTS.md → 让 Agent 每次启动读到这条规则
  3. 加入 CLI Hook → 写一个 PreToolUse hook，在 Agent 写入文件前用正则扫 sk-[a-zA-Z0-9]{20,}，匹配到就直接 exit 2 拦截，返回 “检测到硬编码 API Key，请改用 process.env.SECRET_KEY“

从此以后，任何 Agent、任何文件、任何时候，都不可能再把密钥写进代码。犯一次错 = 永恒免疫。

这个循环的威力在于复利效应：第一周你可能要加 10 条规则，第二周 5 条，一个月后可能一周才遇到一条值得固化的情况。Agent 的”犯错率”会随着时间收敛到趋近于零。

1.4 熵管理（Entropy Management）

软件工程的熵定律：代码放着不管，只会越来越乱。 在 Agent 驱动的项目中，这条定律会被加速放大——因为 Agent 学习代码库时会无差别复现一切模式，包括那些糟糕的模式。

比如你中途手动修了一处临时方案没来得及重构，Agent 扫描到这段代码后，认为这是”约定俗成的做法”，然后在 10 个新文件里照抄了一遍。原本一处技术债，一夜之间变成 11 处。

传统做法是每周五花 20% 时间人工清理，但 Agent 项目的坏模式扩散速度超出了人工清理的能力。OpenAI 的解法是：

  1. 将”黄金规则”编码进仓库 — 用 ast-grep、自定义 ESLint 规则、Import Linter 把好品味机械固化（回到 1.2）
  2. 定期运行后台”园丁 Agent” — 类似 GC 垃圾回收，扫描全仓库找出偏离黄金规则的代码，自动发起小增量重构 PR

例子：import 混乱的自动治理

刚开始项目小，Agent 从 ui/ 层直接 import database/ 层的东西，能跑就行。三个月后这种”能跑就行”的 import 遍布 200 个文件，分层形同虚设。

传统做法：开一个”重构 import 层”的 epic，排期两周，最后不了了之。

熵管理办法：

  1. 团队一条 grimp import lint 规则，CI 不做全量报红（那会炸），只对新 PR 的增量部分进行检测
  2. 同时设一个”园丁 Agent”，每周六凌晨自动跑一次全仓库 import 扫描，把违反分层的 import 自动生成重构 PR，一次修 3-5 个文件，小增量持续还债
  3. 三个月后，存量坏 import 逐批清零，新 import 永远不可能违规

核心思想：不是人的意志力对抗熵增，而是用自动化对抗自动化。 Agent 制造的混乱，由 Agent 自己来收拾。
