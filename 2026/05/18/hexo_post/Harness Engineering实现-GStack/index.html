Harness Engineering 实现 — GStack

概述

GStack 是 Y Combinator CEO Garry Tan 开源的一套 AI 辅助软件工程工作流。它将 Claude Code 变成一个虚拟工程团队，包含 47 个专业角色（CEO 评审、工程经理、设计师、QA、安全官、发布工程师等），每个角色都是一个 Markdown 技能文件（SKILL.md）。

  • 源码仓库：github.com/garrytan/gstack
  • 本地路径：/Users/shaver/claude_projects/GStack
  • 许可证：MIT，永久免费

核心理念

Garry Tan 用 GStack 实现了惊人的生产率飞跃：

2026 年 vs 2013 年：逻辑代码变更量是当年的 ~810 倍，2026 年前 4 个月产出已经是 2013 年全年的 240 倍。一个人 + AI 做到了以前 20 人团队的产出。

三大原则（Builder Ethos）

  1. Boil the Lake（做完整） — AI 让”做完整”的边际成本趋近于零。90% 和 100% 之间就差几秒，永远选 100%。
  2. Search Before Building（先搜索再构建） — 三层次知识：经典方案 → 流行趋势 → 第一性原理。避免重新发明轮子，但也不盲从。
  3. User Sovereignty（用户主权） — AI 建议，人类决策。这个规则高于一切。

参考源码文件：ETHOS.md，ARCHITECTURE.md

源码结构

关键架构：浏览器守护进程

首次调用 3s 启动 Chromium，后续每次 ~100-200ms。技术选型：Bun 编译为单二进制文件（58MB），内置 SQLite 用于读取浏览器 Cookie 数据库。

详见源码：ARCHITECTURE.md 第 5-50 行，browse/src/ 目录

安装指南

前置要求

  • Claude Code
  • Git
  • Bun v1.0+

方式一：30 秒安装（个人使用，推荐）

安装分两步：第一步下载并运行安装脚本，第二步在项目 CLAUDE.md 中注册技能。

Step 1：下载并安装

在 Claude Code 会话中粘贴以下命令，Claude 会自动完成安装：

这条命令逐段拆解：

命令片段 作用
git clone --single-branch --depth 1 浅克隆，只拉最新版本（不要完整历史，快）
https://github.com/garrytan/gstack.git 从 Garry Tan 的 GitHub 仓库下载
~/.claude/skills/gstack 安装到 Claude Code 的全局技能目录，所有项目共用
&& cd ~/.claude/skills/gstack 进入刚下载的目录
&& ./setup 运行安装脚本

./setup 脚本具体做了什么：

  1. 检测你机器上装了哪些 AI agent（Claude Code / Codex / Cursor / Kiro / Hermes 等 10 种）
  2. 为 ~/.claude/skills/ 下创建每个技能目录（如 qa/、review/），里面的 SKILL.md 以符号链接指向 gstack 源码
  3. 编译浏览器二进制文件到 browse/dist/
  4. 安装 Bun 依赖
  5. 询问你偏好短命令（/qa）还是命名空间命令（/gstack-qa），记住偏好供后续升级使用

执行完后，47 个 / 命令就已经注册到 Claude Code 的技能系统里了。但 Claude 还不知道它们存在——需要 Step 2。

Step 2：在项目 CLAUDE.md 中注册技能

Claude Code 每次启动时会读取项目根目录的 CLAUDE.md 作为上下文。你需要手动添加以下段落，告诉 Claude 有哪些技能可用：

这段配置逐行解释：

行 作用
Use /browse from gstack for all web browsing 告诉 Claude：所有网页浏览用 GStack 的 /browse，不要再开别的浏览器工具
Never use mcp__claude-in-chrome__* tools 禁止 Claude 使用 Chrome MCP 工具（GStack 自带更强的浏览器，避免冲突）
Available skills: /office-hours, ... 列出所有可用技能名称，让 Claude 知道有什么 / 命令可以调用

为什么需要这一步？ Claude Code 的技能发现机制不是自动扫描 ~/.claude/skills/ 目录的——它依赖 CLAUDE.md 中的声明来知道有哪些技能。没有这段配置，Step 1 等于白做。

最佳实践路线

核心理念

GStack 的精髓在于流程化——每个技能的输出自动喂给下一个技能，形成完整的软件工程流水线：

场景一：完整新功能开发（黄金路线，7 步）

前提： 你有一个新功能想法，需要从零到生产。

步骤 命令 示例交互 产出
1. 思考 /office-hours “我想做一个日历简报 App” → Claude 追问：痛点是散落的日历事件还是简报格式不好？可能重新定义为”私人 AI 参谋长” 设计文档
2. 规划 /autoplan 自动运行 CEO→设计→工程→DX 四轮评审，每轮只看 Agree/Disagree 的关键决策 评审通过的详细实施计划
3. 确认 退出 plan mode 确认计划后 Claude 开始实现 —
4. 构建 Claude 自动写代码 一次性输出完整实现（典型：2400 行/11 文件/8 分钟） 完整代码
5. 评审 /review 找生产级 bug，自动修简单问题（[AUTO-FIXED]），复杂问题标出让你确认（[ASK]） 修复后的代码
6. 测试 /qa https://staging.xxx.com 开真实浏览器，点击流程，找到 bug → 自动修 → 写回归测试 → 再验证 通过测试的代码 + 回归测试
7. 发布 /ship 同步 main 分支 → 跑测试 → 审计覆盖率 → 推送 → 创建 PR GitHub PR

场景二：小需求 / 修 Bug（3 步）

适合：修复一个已知 bug、改个文案、调整样式等。不需要规划和 QA。

场景三：方案验证（4 步）

适合：有方案但不确定对不对，需要架构审查但不需要完整的设计评审。

场景四：安全检查（1 步）

OWASP Top 10 + STRIDE 威胁建模。每天可跑（8/10 置信度门槛），每月跑一次全面扫描（2/10 门槛）。

场景五：设计探索（5 步）

适合：需要设计前端页面但不确定长什么样。AI 生成多方案→浏览器对比→迭代→输出生产级 HTML。

场景六：技术写作（补充流程）

/ship 默认会触发 /document-release，确保 README、ARCHITECTURE、CHANGELOG 等与代码同步。

GStack 与 Harness Engineering 的关系

GStack 不是传统意义上的 “Harness Engineering 框架”。它在三个层面上实现了 Harness 理念：

层面 GStack 中的实现 源码位置
Browser Harness (浏览器操控层) Playwright + Chromium 守护进程，持久化会话，100ms 命令延迟 browse/src/
Workflow Harness (工作流编排层) 47 个 SKILL.md，每个是一个专业工程角色，输入输出彼此衔接 各技能目录下的 SKILL.md
Agent Harness (Agent 协调层) /pair-agent 安全地让多个 AI Agent 共享同一浏览器会话 pair-agent/SKILL.md

核心洞察： GStack 的”Harness”就是那套使单个开发者能指挥整个 AI 团队的流程骨架。浏览器是可插拔的——你有自己的 browser harness 或 BAI 方案，依然可以使用 GStack 的其他 46 个技能。两者形成互补而非绑定关系。
